셔틀버스 GPS 운행 이력관리

화면 기획서 및 로우피델리티 와이어프레임

화면 개요

특정 운행일과 시간 범위를 통하여 셔틀 차량의 GPS 로그 이력을 조회·분석하고 문제 발생 시 개별 로그를 지도에서 확인하는 관제용 화면입니다.

마커와 데이터가 많은 점을 고려하여 테이블에서 정보를 선택하고 지도로 위치를 확인하는 방식(테이블 위주 탐색)을 핵심 구조로 삼습니다.

화면 구성 요소

  • 1

    상단 컨트롤 바 (필터)

    조회를 위한 항목 지정 영역입니다. 단일 운행날짜, 시간 범위, 특정 차량을 선택하고 [조회] 버튼을 클릭하면 아래 지도와 테이블이 갱신됩니다.

  • 2

    지도 영역 (화면의 40%)

    실제 카카오맵이 렌더링될 영역. 선택 기간 내 GPS 포인트를 점선 형태의 폴리라인 경로 및 포인트 마커로 표시합니다.

  • 3

    마커 및 경로 피드백

    마커의 수가 많을 수 있으므로 기본적으로는 작게/연하게 표시하고, 테이블에서 선택한 로그의 좌표만 크고 강조된 스타일로 변환과 동시에 해당 위치로 지도 중심을 이동시킵니다.

  • 4

    로그 데이터 테이블 (화면의 60%)

    사용자 메인 인터랙션 영역으로, 검색 조건의 수 만큼 로그를 리스트 형식으로 출력합니다. (정렬 기능 및 로그번호, 차량ID, 시간, 속도 기본 노출)

  • 5

    테이블 중심 양방향 선택

    테이블의 특정 로그(행) 데이터에 Hover 또는 클랙했을 때 지도 상에 해당하는 마커 모양이 강하게 피드백되도록 연동해야 합니다.

주요 지표 및 속성 도출 방식

화면에 표시되는 요약 정보 및 확장 컬럼들은 Raw GPS 단일 로그로부터 다음과 같은 방식으로 가공되어 노출됩니다.

  • 1) 속도 데이터 (개별, 평균, 최대)

    도출 방식: 단말기에서 전송된 실시간 GPS 단일 로그의 speed 속성값을 기준(단위: km/h 변환)으로 합니다.

    • 개별 로그 속도 표시: 테이블 우측 '속도' 컬럼에 수치 표시. 특정 기준(예: 60km/h 초과 시)을 넘어가면 빨간색 폰트 및 경고 아이콘(⚠️)으로 시각적 차별화.
    • 최대 속도: 선택된 기간 내 로그 집합 중 Max(speed)
    • 평균 속도: Sum(speed) / 유효 로그 수 (신호 대기 등 0km/h인 상태를 평균 속도 연산에 포함할지 여부는 서버 단독 정책으로 설정)
    *스마트폰 관제 단말기 (Android/iOS) 유의사항
    스마트폰의 네이티브 Geolocation API에서도 speed 값은 기본 반환되지만, 터널/지하차도나 빌딩 숲 등 음영 지역에서 튀는 값(Outlier, 예: 갑자기 200km/h)이 들어올 확률이 큽니다. 따라서 서버 로직에서 (1) 신뢰도 필터링(Accuracy Threshold) 및 (2) 비정상 속도값 Smoothing(평활화) 보정 처리 후 화면에 제공해야 합니다.

  • 2) 확장 컬럼: 상태 (운행 / 정차 / 장기정차)

    도출 방식: 일련의 연속된 로그 값들의 속도와 좌표 변화폭을 기반으로 연산.

    • 운행: 현재 로그 속도가 기준치 이상 (예: speed > 3km/h)
    • 정차: 속도가 연속해서 0이거나 기준치 이하 (예: speed <= 3km/h)
    • 장기정차: [정차] 상태가 N분 이상 지속되는 경우 상태를 치환시켜 경고 알림 연계.

  • 3) 확장 컬럼: 체류 시간

    도출 방식: 현재 로그가 '정차' 혹은 '정류장 근접' 상태라고 판별되었을 때의 지속 시간 산출.

    • 차량 좌표가 반경 X 미터 내에 머물기 시작한 시점 T(start) 저장.
    • 이후 X 미터 범위를 벗어난 시점 T(end)와의 시간차(Minutes)를 계산하여 노출.

  • 4) 확장 컬럼: 위치명 (Reverse Geocoding)

    도출 방식: GPS 로그의 `Lat`, `Lng` 좌표를 사용하여 문자열 형태의 주소나 정류장 이름 변환.

    • 마스터 매칭: 사전에 DB에 등록된 정류장 좌표들 반경 30m 반경 안에 들어왔을 시 해당 [정류장명] 매칭 표시.
    • API 호출: 좌표와 매칭되는 정류장이 없을 시, 카카오맵의 리버스 지오코딩 API를 사용해 [간략한 도로명 주소] 표시 (단, 비용 문제로 행 클릭/상세 확인 시에만 On-Demand로 동작 권장).

⚠️ 정부 및 정책 처리 (개발/디자인 고려사항)

  • GPS 음영 지역 (Drop): 지하차도/터널 등 신호 끊김 시 마지막 좌표와 연결된 좌표 사이를 직선 점선 보간 또는 별도 색상으로 잇고 '데이터 누락구간'으로 명시합니다.
  • GPS 튐(Noise) 보정: 1초만에 과도한 거리를 이동한 물리적 불가능 값은 비정상 이상치(Outlier)로 판단, 화면 및 평균 계산에서 제외하는 백엔드 전처리가 필요합니다.
  • 정차 및 운행 상태 기준: 지정 정류장 반경 30m 이내에서 속도가 0km/h일 때만 '정류장 체류'로 판정하며, 일반 도로에서 0km/h일 경우 단순 '교통체증 대기'로 분리합니다.
  • 대용량 데이터 렌더링 (프론트 성능): 1초 단위 로그 수집 시 장시간 조회하면 수만 개의 Row가 생성됩니다. 브라우저 프리징을 막기 위해 테이블은 가상 스크롤(Virtual Scrolling)을 이용하고, 맵 마커는 클러스터링 혹은 특정 축척비례 생략(Simplification) 적용이 필수입니다.
